# Project Skill Notes

## 通用注意事项

1. 行间公式中的左右大括号统一使用 `\left\lbrace` 和 `\right\rbrace`，不要使用其他写法（例如 `\left\{`、`\right\}`）。
2. 本文件用于记录项目范围内的通用写作与排版约定；新增规则时请保持表述简洁、可执行。
3. 其他公式中使用的宏，统一参考 `docs/javascripts/mathjax.js` 中的定义，不要自行引入不在该文件中的宏写法。
4. 提示（Hint）统一使用可收放语法 `??? hint "提示"`（默认收起），用于给思路引导但不直接给完整证明；视觉上使用与项目既有 hint 一致的轻字体、浅背景框样式。该语法前需要保持与正文的空行分隔，以免被解析器误认为是段落内的内容，导致样式异常。
5. 额外引用且不要求学生证明的定理/引理，统一使用非收放语法 `!!! theorem "定理（可直接使用）"`（引理可改标题为“引理（可直接使用）”）；其视觉样式应与 hint 保持同一背景色与弱化风格，用于与正文区分但不做强强调。该语法前需要保持与正文的空行分隔，以免被解析器误认为是段落内的内容，导致样式异常。
6. 若题目内容中包含算法步骤描述，也需要使用与定理相同的背景框。即中英文分别使用 `!!! theorem "算法 算法名"` 和 `!!! theorem "Algorithm 算法名"`（例如 `!!! theorem "算法 MH-Alt"`）。为了更贴近 LaTeX 伪代码风格（无点、有间隔、输入与代码分离），建议的排版格式为：在“输入”或“输出”行后加一空行，然后在算法各行前使用纯数字加等宽空格（如 `1 &nbsp;&nbsp;`）替代原生的 Markdown 有序列表。同时，在每一行的末尾添加两个空格实现软换行。若遇循环/条件结构，可通过继续累积 `&nbsp;` 实现不同层级的缩进（例如 `3 &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;`）。以此保证它既具有原生伪代码的干练缩进外观，又继承定理背景框的主题样式。
7. 在多行数学环境（如 `\begin{aligned}`、`\begin{cases}` 等）中，**不要使用 `\\` 换行**，统一使用 `\cr`。原因是 Markdown 解析器可能在 `\\` 到达 MathJax 之前错误地转义或吞掉反斜杠。
8. 避免公式与文字粘连：在文字段落中的行内公式 `$xxx$` 或者 `\(xxx\)` 的前后，如果紧跟的是中英文文字或数字，需要主动加上一个空格，以免网页渲染出来公式和文字挨得太近；如果公式前后跟的是标点符号（如中英文逗号、句号等），则不需要加空格。注意，避免误加空格产生 `$ xxx$`，`$ xxx $` 或者 `$xxx $` 这样的格式。
9. 列表分离：在 Markdown 排版中，如果遇到一个无序列表（如 `- `）紧接着一个有序列表（如 `1. `）被解析器误识别合并成同一个列表嵌套的情况，需要在两个列表之间插入一行带有相同缩进级别的空注释 `<!-- -->` 来强制打断列表解析。

## 题目迁移与排版规范（TeX 到 Markdown）

1. 结构与标题格式：每道题目（无论是 Exercises 还是 Problems）需采用三级标题 `### 中文标题 (English Title)`。
2. 语言标签块结构：必须使用带有缩排的 PyMdown 选项卡语法（Tabbed syntax）来组织中英双语内容。
    - `=== "中文"`
    - 所属题目内容统一缩进 4 个空格。
    - 确保迁移过来的公式排版按照“通用注意事项”进行了修正（大括号 `\left\lbrace`、`\right\rbrace` 及换行符 `\cr`）。
    - 将 TeX 中的特殊环境转换为 Markdown admonitions（如 `??? hint "提示"`，或 `!!! theorem "定理（可直接使用）"`），内容同样要求维持相应的缩进级别。

## 英文翻译要求

1. 结构对齐：在中文内容块的后方，紧接着建立等效格式的英文标签块：
    - `=== "English"`
    - 其下的所有内容同样需要缩进 4 个空格。
2. 内容对齐：**保证意思和原中文题完全一样**，公式的写法及行文格式也必须与中文部分一模一样，不得缺少信息或擅自额外增加说明内容。
3. 语言风格：要求简洁、严谨，符合英文数学与计算机类讲义的行文习惯。确保语言自然、通顺，**绝对避免明显的生成式 AI 翻译口吻**，切忌由于翻译繁杂冗长导致行文啰嗦。
4. 对应环境修改：提示和定理块也要对应翻译（如相应的替换为 `??? hint "Hint"` 和 `!!! theorem "Theorem (Can be used directly)"`），保持结构对称。

## 解答引入语与答案文件创建规范

1. 引入语句 (`--8<--`) 的使用：在中英文的 Tab 层级结束之后（通常取消前置缩进），每一道题的末尾都要利用 Snippets 语法引入外部答案文件。格式为：
    - `--8<-- "solutions/chapter_XX/exercises/exercise_题目对应的英文简写.md"`（用于 Exercises）
    - `--8<-- "solutions/chapter_XX/problems/problem_题目对应的英文简写.md"`（用于 Problems）
2. 答案文件命名关联：要使被引入的 Markdown 答案文件名同题目的英文标题（English Title）存在简单直观的对应关系。通常采用全小写、空格转下划线的方式命名。例如标题中有 distinguishing cost，可以用 `problem_distinguishing_cost.md` 分配。
3. 同步创建空白文件：在使用 `--8<--` 补充引入语句的同时，必须在对应的路径下（如 `docs/solutions/chapter_XX/.../`）**创建相应的 `.md` 答案文件**。初始化时这些文件的内部保持空白即可，方便读者后续在 GitHub 提 Pull Request 追加自己的解答。
